Track 9 — Combat state v2 (done; tempo control, not just hitstun; cargo verify pending)
  Current v1 says "advantage" iff opponent is in hitstun → which is
  actually "true combo in progress", not advantage. v2 layers signals
  to capture the looser definition: who controls the tempo of this
  exchange right now? Build it incrementally; each layer composes
  on top of the previous one without changing the CombatState enum.

  Implementation shape (settled while planning):
    - `classify_frame_v2` / `classify_frame_layered` is pure: takes a
      `PlayerFrameSignals` per side (action_state, offstage, on_ledge,
      frames_since_hitstun) plus `CombatV2Config`, returns CombatState.
      Easy to unit-test with table-driven cases.
    - `compute_analysis_1v1` sweeps frames, maintains per-player
      "frames since the last hitstun ended" counters across the loop,
      looks up `stage_bounds(stage_id)` once for offstage detection,
      and feeds each frame's signals to the pure classifier.
    - `classify_frame` (v1) stays around as the hitstun-only helper —
      both for backward compat and as the inner-loop call from v2.

  [x] 9a: Hitstun-tail recency window.
       Implemented as `CombatV2Config::hitstun_tail_frames` (default 30
       frames). The sweep maintains `frames_since_hitstun: Option<u32>`
       per player, set to `None` while in hitstun, `Some(0)` the frame
       hitstun ends, incrementing thereafter. Layer fires when the
       counter ≤ tail.
  [x] 9b: Offstage overlay.
       `compute_analysis_1v1` looks up `stage_bounds(stage_id)` once
       and per-frame asks `is_offstage(x, y)`. Stages outside the legal
       pool return `None` and the layer silently disables. Reads
       `position.{x,y}.get(idx)` from peppi's f32 columns.
  [x] 9c: Ledge-state overlay.
       `LEDGE_STATE_MIN`/`MAX` constants (0xFC..=0x107) bracket every
       `CLIFF_*` action state — catch, wait, climb, attack, escape,
       jump. `is_on_ledge(state)` is the predicate; the edge layer
       merges offstage-OR-ledge into one "stuck on the perimeter"
       check.
  [x] 9d: Priority collapse rules.
       Documented in the module docstring: hitstun > recency > edge >
       neutral. Hitstun ties → Trade; recency ties → more-recent-hit
       wins, exact tie → Trade; edge ties → fall through to neutral
       (mutual edgeguard isn't anyone's advantage).
  [x] 9e: Tests — eleven-case suite covers each layer in isolation
       (`is_in_hitstun`/`is_on_ledge` boundaries, hitstun-only,
       recency-only, edge-only) and the composition rules
       (hitstun overrides everything, recency overrides edge,
       tail-frames=0 collapses to v1).
  [ ] 9f: cargo check + cargo test after classifier v2.

Track 11 — Replay analysis sidecar cache (done; cargo verify pending)
  Goal: stop re-parsing peppi every time the user clicks "View". Today
  `load_viewer` calls `parse_replay_analysis` on every entry into the
  viewer page — ~1s for a 5-minute replay, climbing as the v2 layered
  classifier does more per-frame work. The fix is a versioned sidecar
  cache, NOT a DB table — see the design discussion below.

  **Policy: persistent.** Frame-derived classifications (combat states,
  eventually positional snapshots) are tens-to-hundreds of KB per
  replay — small enough that "keep everything the user has ever
  viewed" is realistic at hundreds of MB total. Defaults on the order
  of:
    - `max_bytes = 500 MB` (only kicks in for power users; LRU by
      mtime when triggered)
    - `prune_on_drop = false` (Drop is a no-op; entries survive app
      restarts, which is the whole point — the second time you open
      a viewer it's instant)
    - No `max_entries` cap (the byte budget is the only ceiling).
  This is the opposite of the video cache (10b) on every axis, which
  is exactly why the two caches share storage primitives but pick
  their own policies.

  Why a file cache, not a DB column / table:
    - Per-frame data is read in the exact shape it was written ("give
      me the full vector for game X"), so it's a blob, not a relation.
    - The combat-state vector changes every time we tune the v2
      classifier (recency window, priority rules, future overlays). A
      DB column would need a migration on every algorithm tweak; a
      versioned sidecar just bumps a `classifier_version` int and
      regenerates lazily on next view.
    - Source of truth always remains the .slp itself. The cache is
      pure derivation; deletion is a no-op for correctness.
    - Track 10b is already building a file cache for MP4s — this
      reuses the same storage module with a different blob format.

  Cache layout:
    <cache-root>/analysis/<slp-hash>.bincode
        Versioned `CachedAnalysis { version, config_hash, analysis }`
        where `analysis` is the existing `combat::ReplayAnalysis`.
    <cache-root>/video/<slp-hash>.mp4
        Track 10's MP4. Same root, sibling subdir.

  [x] 11a: Generic byte-keyed file cache (shared core).
       Module is `stats-melee/src/file_cache.rs`. Built jointly with
       Track 10b's video cache and Track 11's analysis cache — neither
       owns this module, both wrap it with their own policy.
       API surface:
         FileCacheConfig {
             max_bytes: u64,
             max_entries: Option<usize>,
             prune_on_drop: bool,
         }
         FileCache::open(root: PathBuf, config: FileCacheConfig)
             -> Result<FileCache>
         FileCache::lookup(&self, key: &str) -> Option<PathBuf>
         FileCache::path_for_write(&self, key: &str) -> PathBuf
         FileCache::finalize(&mut self, key: &str) -> Result<()>
         FileCache::evict_to_budget(&mut self) -> Result<usize>
         FileCache::clear(&mut self) -> Result<usize>
         impl Drop for FileCache (calls clear if config.prune_on_drop)
       LRU by mtime (touch on read), eviction triggered on finalize
       OR on app close via the prune_on_drop flag.

       The two consumers pick their own config:
         - VideoCache: `{ max_bytes: 2 GB, max_entries: Some(5),
                          prune_on_drop: true }`
         - AnalysisCache: `{ max_bytes: 500 MB, max_entries: None,
                             prune_on_drop: false }`
       Each lives under its own subdir of `ProjectDirs::cache_dir`

         4. Else, recompute via parse_replay_analysis, then write

  [x] 11d: slp_hash decision — content vs. path.
       Path-based: cheap (no I/O), but not portable across machines
       and breaks when the user moves their replay folder.
       Content-based (SHA-256 of file): one extra full read per
       cache miss, but stable and dedup-friendly (re-ingesting the
       same .slp under a renamed path hits the same cache entry).
       Recommendation: content-based, computed once at ingestion time
       and stored on the `game` row as a new `content_hash` column.
       The viewer then never re-hashes — it pulls the precomputed
       hash off the row. Adds one migration; pays for itself the
       first time a user moves their replay folder.
